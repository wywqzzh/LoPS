# 项目研究摘要

**项目：** LoPS  
**领域：** 科研 Python 脚本重构与一致性验证  
**研究日期：** 2026-05-03  
**置信度：** MEDIUM

## 摘要

LoPS 的核心不是构建一个通用软件平台，而是建立一套可靠的科研脚本重构流程。它的成功标准是：每轮都能先理解外部目标脚本，再得到用户确认的重构方案，然后把功能迁移到本仓库的模块、脚本和数据结构中，并用旧新输出对比证明结果没有变化。

技术上应保持保守：遵循目标脚本原有运行环境，避免为了现代化而改变依赖或数值行为。项目级结构可以逐步向现代 Python 包靠拢，但每轮重构首先要保护输入、输出、路径、随机性和依赖版本这些会影响科研结果的因素。

最大风险是“看起来重构成功，但科研结果已经改变”。因此路线图必须把原始行为证据、方案确认和一致性验证放在核心路径上。

## 关键发现

### 推荐技术栈

- Python：沿用用户指定科研环境，避免无意改变数值行为。
- `pyproject.toml`：后续需要正式包配置时采用 PyPA 推荐的现代 Python 项目配置方式。
- pytest：用于临时目录、环境替换、输出捕获和验证脚本。
- NumPy：保留科研脚本常见数值栈，并显式处理随机 seed。
- `subprocess.run`：用于封装旧脚本和新脚本的运行对比。

### 预期功能

**必须具备：**
- 接收目标脚本、环境和数据来源。
- 记录原始功能、输入输出、执行流程和依赖。
- 判断模块化是否必要。
- 修改前产出方案并等待确认。
- 在 `src/LoPS`、`script`、`data` 中实现确认后的结构。
- 用相同输入和 seed 对比旧新输出。
- 对未固定随机过程使用 `src/LoPS/temp` 临时旧副本验证，验证后清理。

**后续增强：**
- 自动生成验证模板。
- 环境快照记录。
- 输出差异诊断。

**暂缓：**
- 批量扫描外部项目。
- GUI 或完整任务平台。
- 多语言科研脚本支持。

### 架构方式

LoPS 应采用分层流程：任务接收、原始分析、方案设计、模块实现、运行入口、验证对比和结果记录。正式模块只放在 `src/LoPS`，运行入口放在 `script`，整理后的数据放在 `data`，临时旧代码只短暂放在 `src/LoPS/temp`。

### 关键风险

1. **输出不一致：** 先建立旧输出基准，再做新实现对比。
2. **未理解就迁移：** Phase 2 必须完整记录入口、I/O、依赖和随机性。
3. **随机过程不可比：** 旧临时副本和新实现使用同一 seed。
4. **数据路径语义变化：** 迁移数据前先记录原始读取规则。
5. **绕过确认门：** Phase 3 结束前不得进入实现修改。

## 对路线图的影响

### Phase 1：项目骨架与任务接收契约

**理由：** 后续每轮重构都需要统一输入格式和目录规则。  
**交付：** 任务接收信息、目录约定、初始记录模板。  
**避免：** 每轮都重新解释工作流。

### Phase 2：原始行为分析与证据采集

**理由：** 没有旧行为证据就无法安全重构。  
**交付：** 原始脚本分析记录、依赖记录、输入输出基准。  
**避免：** 未理解脚本就模块化。

### Phase 3：重构方案与确认门

**理由：** 用户明确要求确认后再执行重构。  
**交付：** 模块边界、接口、数据路径、验证方案。  
**避免：** 方案未确认就改代码。

### Phase 4：模块化实现与运行入口

**理由：** 确认后才能把功能落入 `src/LoPS`、`script` 和 `data`。  
**交付：** 新模块、运行脚本、整理后数据路径。  
**避免：** 模块和脚本边界混乱。

### Phase 5：一致性验证与收尾记录

**理由：** 科研重构必须以结果一致为最终证明。  
**交付：** 旧新输出对比、随机性验证、临时代码清理、完成记录。  
**避免：** “能运行”但结果不一致。

## 需要后续验证的空白

- 第一轮真实目标脚本尚未提供，因此模块命名、数据子目录和验证输出格式还不能定死。
- 当前仓库没有测试框架文件，是否引入 pytest 配置应在第一轮需要验证时决定。
- 如果外部脚本依赖专门环境，LoPS 需要记录并复用该环境，而不是强行迁移依赖。
