# Plan 03-01: 建立过程一致性基线和核心类型契约

<objective>
在任何算法实现改动前，先建立 Phase 3 的内部数据类型契约和过程一致性测试基线。该计划只做低风险的类型辅助与测试锁定，不改变 `GrammarLearner.learn()` 的候选评分、迭代选择、skip-gram 或正式输出语义。
</objective>

<context>
用户明确要求本阶段不是结构层小修补，而是从顶层算法设计向下优化；同时要求优化后不仅最终输出一致，中间处理过程的重要指标也要一致。因此第一步必须先把当前关键过程固化成测试，避免后续优化只在最终 pickle 层面“碰巧一致”。
</context>

<design_contract>

## 核心类型设计

```python
GrammarToken = tuple[str, ...]

@dataclass(frozen=True)
class ParsedSequence:
    """保存一次最长匹配解析得到的全部派生信息。"""
    tokens: list[GrammarToken]
    token_strings: list[str]
    token_counts: dict[str, int]
    token_probabilities: dict[str, float]
    token_time: dict[str, float]
    position_grammar: list[str]
```

- `GrammarToken` 只在核心算法内部使用，例如 `("G", "L")`。
- `token_strings` 和 `position_grammar` 是边界展示或验证所需的字符串表示。
- `ParsedSequence` 不包含候选筛选结果，也不承载 pair posterior，避免变成错误的 `TransitionStats`。

## 函数接口设计

```python
def parse_token_string(token: str) -> GrammarToken
def format_grammar_token(token: GrammarToken) -> str
def combine_grammar_tokens(parent: GrammarToken, child: GrammarToken) -> GrammarToken
def grammar_tokens_share_base(left: GrammarToken, right: GrammarToken) -> bool
```

这些函数是 tuple token 的最小接口；原有字符串辅助函数可作为边界包装保留，后续计划逐步把核心调用迁移到 tuple 接口。
</design_contract>

<threat_model>

| Threat | Severity | Mitigation |
|--------|----------|------------|
| 过程测试创建太晚，无法确认重构前指标 | HIGH | 本计划先在当前实现上写过程测试并通过，再进入后续算法改动。 |
| `ParsedSequence` 被误用成候选统计对象 | HIGH | 数据类只保存解析和概率派生信息，不保存 pair posterior、候选过滤或评分。 |
| 测试数据误放入 `.planning` | MEDIUM | 若需要生成过程追踪文件，只能写入 `data/generate_grammar/process-traces/`；优先使用测试代码中的小型内联期望值。 |
| 为类型契约引入复杂类层级 | MEDIUM | 只使用类型别名、dataclass 和少量纯函数，保持 KISS。 |
</threat_model>
