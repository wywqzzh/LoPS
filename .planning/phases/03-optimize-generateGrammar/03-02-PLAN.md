# Plan 03-02: 合并解析与概率统计

<objective>
把当前分散在 `_parse_longest()` 和 `_parse_probabilities()` 中的重复解析过程合并为一次 `ParsedSequence` 构建流程，减少重复 token 拆分和重复最长匹配扫描，同时保持解析序列、位置级 grammar、概率、频次和时间占比完全一致。
</objective>

<context>
当前实现每轮学习会先做最长匹配解析，再为了概率和时间占比重复处理 token。用户同意引入 `ParsedSequence`，但强调必须保证 parse 结果和概率统计与源代码一模一样。本计划只优化解析派生流程，不触碰候选评分顺序、不做 TransitionStats、不改变 skip-gram。
</context>

<design_contract>

## 顶层算法变化

当前流程：

```text
grammar -> _parse_longest(sequence, grammar)
grammar -> _parse_probabilities(parsed_tokens)
```

目标流程：

```text
grammar -> _build_parsed_sequence(sequence, grammar) -> ParsedSequence
ParsedSequence -> parsed token list / position grammar / probability / time
```

## 函数接口设计

```python
def _build_parsed_sequence(
    self,
    token_sequence: Sequence[str],
    grammar: Mapping[str, Sequence[str]],
) -> ParsedSequence
```

- 输入仍接收当前公开边界的字符串序列，函数内部立即转换为 `GrammarToken`。
- 匹配逻辑必须保留当前“按 grammar key 长度降序进行最长匹配”的语义。
- `_parse_longest()` 和 `_parse_probabilities()` 可以作为兼容测试入口保留，但内部应委托给 `ParsedSequence`，不能再各自重复实现一套解析。
</design_contract>

<threat_model>

| Threat | Severity | Mitigation |
|--------|----------|------------|
| 合并解析后改变并列 chunk 匹配顺序 | HIGH | 过程测试必须覆盖相同长度候选和复合 token 候选；保持当前排序规则。 |
| 时间占比浮点顺序变化导致旧新不一致 | HIGH | 概率和时间占比使用与当前实现一致的分母和插入顺序，测试使用 exact equality 或必要时记录 IEEE 浮点同值。 |
| 为删除 `position_grammar` 过早改动适配器 | MEDIUM | 本计划先保证解析派生一致，正式输出字段调整放到 03-05。 |
| tuple/string 来回转换污染核心逻辑 | MEDIUM | 只在输入输出边界格式化字符串，核心匹配用 tuple。 |
</threat_model>
