# Plan 03-04: 重整 GrammarLearner.learn 主循环

<objective>
在保持候选评分语义完全不变的前提下，把 `GrammarLearner.learn()` 的主循环改成更清晰的顶层数据流：每轮只解析一次、使用数组化离散数据、按当前顺序计算候选得分、选择 chunk、更新 grammar，并删除无效参数 `candidate_ratio_min`。
</objective>

<context>
用户特别指出不能先做 `TransitionStats`，因为当前转移统计来自含先验的 `pair_posterior`，不是纯数据频次。用户还要求暂不考虑基于 pair frequency 的性能优化。因此本计划的核心是“重排代码结构以更直接表达当前算法”，不是改变候选生成和选择策略。
</context>

<design_contract>

## 顶层流程设计

```text
初始化 grammar 和 active_tokens
for iteration in max_iterations:
    parsed = _build_parsed_sequence(...)
    for parent, child in ordered_candidate_pairs:
        discrete = _organize_discrete_data(parsed, parent, child, ...)
        score_without_parent = bd_score(...)
        score_with_parent = bd_score(...)
        pair_posterior = posterior_from_current_scoring(...)
        candidate_rows.append(...)
    best_candidate = current_selection_rule(candidate_rows)
    if stop: break
    update grammar / active_tokens / traces
```

## 函数接口设计

```python
def _score_candidate_pair(
    self,
    parsed: ParsedSequence,
    parent_token: str,
    child_token: str,
    state_values: np.ndarray,
) -> CandidateScore

def _select_next_chunk(
    self,
    candidate_scores: Sequence[CandidateScore],
) -> CandidateScore | None
```

- `CandidateScore` 可以是轻量 dataclass 或现有 dict，但必须清晰命名 `score_without_parent`、`score_with_parent`、`pair_posterior`、`ratio`。
- `_select_next_chunk()` 不改变当前排序、过滤和停机语义。
- 不引入 TransitionStats，不做提前 frequency prefilter。
</design_contract>

<threat_model>

| Threat | Severity | Mitigation |
|--------|----------|------------|
| 抽函数后候选遍历顺序改变 | HIGH | 过程测试记录每轮候选顺序、best chunk、ratio 和 KL 历史。 |
| 删除参数时破坏配置构造或测试 | MEDIUM | 同步更新 `GrammarLearningParams`、测试和运行脚本，确认无引用残留。 |
| 为清晰度引入过多类 | MEDIUM | `CandidateScore` 若使用 dataclass，只保存一行评分结果，不建立候选管理框架。 |
| 主循环改写后最终验证通过但过程漂移 | HIGH | 新增 per-iteration trace 测试，锁定代表性输入的中间指标。 |
</threat_model>
